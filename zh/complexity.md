# 复杂度速查表

> 中文版。English: [complexity](../complexity.md)

这些操作的代价是你应该烂熟于心的，这样你就能不假思索地说出「那次查找是
O(1)，所以整体是 O(n)」。下面的每个数字都是针对 CPython（参考解释器）的。
「平均」指的是摊还或在正常哈希下的期望；在会造成麻烦的地方标注了最坏情况。

最常见的面试错误，是使用了一个某个看似无害的操作其实是 O(n) 的结构：在列表头部插入、
检查列表的成员归属，或通过反复拼接来构建字符串。把这些学会，你就不会一不小心写出
O(n^2) 的解法。

## list（动态数组）

| 操作 | 复杂度 | 说明 |
|---|---|---|
| `a[i]`, `a[i] = x` | O(1) | 索引是直接的 |
| `a.append(x)`, `a.pop()` | O(1) 摊还 | append 偶尔会重新分配内存 |
| `a.pop(0)`, `a.insert(0, x)` | O(n) | 所有元素都要移动。改用双端队列 |
| `a.insert(i, x)`, `del a[i]` | O(n) | 移动后面的部分 |
| `x in a` | O(n) | 线性扫描。成员归属请用集合 |
| `a[i:j]`（切片） | O(k) | k = j - i，复制这段切片 |
| `len(a)` | O(1) | 已存储，无需计数 |
| `min(a)`, `max(a)`, `sum(a)` | O(n) | 完整扫描 |
| `a.sort()`, `sorted(a)` | O(n log n) | Timsort，稳定 |
| `a.reverse()`, `a[::-1]` | O(n) | 切片形式还会复制 |
| `a + b` | O(n + m) | 构建一个新列表 |

陷阱：`a.pop(0)` 和 `a.insert(0, x)` 是 O(n)。基于列表构建的队列每次出队都是
O(n)。请抓取 `collections.deque`。

## collections.deque（双端队列）

| 操作 | 复杂度 | 说明 |
|---|---|---|
| `d.append(x)`, `d.appendleft(x)` | O(1) | 两端都很便宜 |
| `d.pop()`, `d.popleft()` | O(1) | 这就是它在队列上胜过列表的原因 |
| `d[i]`（随机访问） | O(n) | 不是数组；访问中间元素需要走过去 |
| `x in d` | O(n) | 线性扫描 |
| `d.rotate(k)` | O(k) | |

任何队列、任何 BFS 前沿、任何滑动窗口最大值的单调缓冲区都用双端队列。当你需要
O(1) 随机访问时不要用它；那是列表的工作。

## set 和 frozenset（哈希集合）

| 操作 | 平均 | 最坏 | 说明 |
|---|---|---|---|
| `s.add(x)`, `s.discard(x)` | O(1) | O(n) | 最坏情况是病态哈希 |
| `x in s` | O(1) | O(n) | 集合存在的理由 |
| `a | b`（并集） | O(len a + len b) | | 新集合 |
| `a & b`（交集） | O(min(len a, len b)) | | 迭代较小的那个 |
| `a - b`（差集） | O(len a) | | |

元素必须是可哈希的（不可变的）：数字、字符串、元组可以；列表和字典不行。当你需要一个坐标集合时，
存储元组 `(r, c)`，不要存列表。

## dict 以及 collections.Counter、defaultdict（哈希表）

| 操作 | 平均 | 最坏 | 说明 |
|---|---|---|---|
| `d[k]`, `d[k] = v`, `del d[k]` | O(1) | O(n) | 哈希 |
| `k in d` | O(1) | O(n) | |
| `len(d)` | O(1) | | |
| 迭代 `d`, `d.items()` | O(n) | | 创建这个视图是 O(1)，遍历它是 O(n) |
| `Counter(iterable)` | O(n) | | 一趟遍历完成计数 |
| `counter.most_common(k)` | O(n log k) | | 内部使用堆 |
| `counter.most_common()`（全部） | O(n log n) | | 完整排序 |

`defaultdict(list)` 或 `defaultdict(int)` 去掉了「先检查键是否存在」的样板代码，
并且不改变复杂度。自 Python 3.7 起，字典保持插入顺序，这偶尔正是整个诀窍所在
（见 LRU 缓存）。

## heapq（基于列表的二叉堆）

| 操作 | 复杂度 | 说明 |
|---|---|---|
| `heapq.heapify(a)` | O(n) | 比 n 次 push 便宜，后者是 O(n log n) |
| `heapq.heappush(a, x)` | O(log n) | |
| `heapq.heappop(a)` | O(log n) | 总是最小值（最小堆） |
| `a[0]`（查看最小值） | O(1) | 别为了看一眼就 pop |
| `heapq.heappushpop`, `heapreplace` | O(log n) | 一次下滤，而不是两次 |
| `heapq.nlargest(k, a)`, `nsmallest(k, a)` | O(n log k) | 底层是一个大小为 k 的堆 |

Python 只有最小堆。要实现最大堆，push `-x`（数字）或一个
`(-priority, item)` 元组。要从已有的数据构建堆，用 `heapify`，
O(n) 完成，不要一个一个 push。

## str（不可变序列）

| 操作 | 复杂度 | 说明 |
|---|---|---|
| `s[i]`, `len(s)` | O(1) | |
| `s + t` | O(n + m) | 每次都构建一个全新的字符串 |
| 在循环中用 `+=` 构建字符串 | O(n^2) | 经典的意外爆炸 |
| `''.join(list_of_str)` | O(总长度) | 构建字符串的正确方式 |
| `sub in s` | O(n * m) 最坏 | 实践中通常接近 O(n) |
| `s[i:j]`, `s.split()`, `s.replace(...)` | O(n) | 都产生新的字符串 |

因为字符串是不可变的，每次「编辑」都要分配内存。如果你在逐个字符地组装输出，
就先收集到一个列表里，最后再 `''.join()`。

## 复杂度阶梯（在 n = 10^6 时每个量级的感受）

| 大 O | 名称 | 大致 |
|---|---|---|
| O(1) | 常数 | 瞬间 |
| O(log n) | 对数 | 约 20 步 |
| O(n) | 线性 | 一趟遍历，没问题 |
| O(n log n) | 线性对数 | 一次排序，到约 10^6 都没问题 |
| O(n^2) | 平方 | 超过约 10^4 就撑不住 |
| O(2^n)、O(n!) | 指数、阶乘 | 只适用于 n 到约 20、约 12 |

把这个和[解题框架](framework/solving-framework.md)里「把 n 当成一种类型来读」的表格
交叉参照：n 上的约束告诉你允许落在哪一行。

## 递归空间的注意事项

如果你的解法用了递归，那么「聪明的 O(1) 空间」这个说法就是错的。每一个递归调用帧都坐在调用栈上，
所以深度为 d 的递归即使不分配任何东西也会用 O(d) 空间。树和 DFS 的解法在栈上是
O(h) 空间（h = 高度），对平衡树是 O(log n)，对退化的树是 O(n)。
真正的 O(1) 空间通常意味着一个使用固定数量指针的迭代解法。只有当你确实做到时，
才说「O(1) 额外空间，迭代式」。
