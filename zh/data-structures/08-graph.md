# 08 - 图

> 中文版。English: [08-graph](../../data-structures/08-graph.md)

图是"事物以及它们之间关系"的通用结构：节点（顶点）由边连接。树和链表只是加了额外规则的图，所以一旦你能表示任意图，你就能建模几乎任何连通性问题。最重要的单一决策是你如何存储边，邻接表还是邻接矩阵，因为它决定了你在其上运行的每次遍历的代价。

![一个图及其邻接表和邻接矩阵](../../assets/graph.svg)

*同一个图的邻接表（O(V + E) 空间，最适合稀疏图）和邻接矩阵（O(V^2) 空间，O(1) 边查询）。*

## 它是什么

图是一组**顶点** V 和一组**边** E，其中每条边连接两个顶点。你必须辨认的几种变体：

- **有向还是无向。**无向边 `(u, v)` 意味着你可以双向走；有向边 `u -> v` 只允许单向。"友谊"是无向的；"关注"或"是……的先决条件"是有向的。
- **带权还是无权。**无权边只说"连着"；带权边携带一个数字（距离、代价、容量）。BFS 处理无权最短路径；Dijkstra 及其同类处理带权的。
- **有环还是无环。**没有环的有向图（DAG）正是拓扑排序所需要的。无向图中的环正是并查集要检测的。

大多数面试里的图是**稀疏的**：边数更接近 O(V) 而非 O(V^2)。一张路网、一张课程先修图、一张社交关注图：每个节点都只与少数几个相连，而非全部。这个事实驱动了下面的存储选择。

## 操作与复杂度

V 是顶点数，E 是边数，`degree(u)` 是从 `u` 出发的边数。代价因表示法而异，这恰恰是表示法重要的原因。

| 操作 | 邻接表 | 邻接矩阵 | 说明 |
|---|---|---|---|
| 空间 | O(V + E) | O(V^2) | 表只存真实存在的边 |
| 添加边 | O(1) | O(1) | 追加到一个列表 vs 设置一个格子 |
| 是否有边 `(u, v)`？ | O(degree(u)) | O(1) | 单点查询矩阵胜出 |
| 遍历 `u` 的邻居 | O(degree(u)) | O(V) | 表只给出真实邻居 |
| 完整遍历（BFS / DFS） | O(V + E) | O(V^2) | 每个邻居列表你只走一遍 |

对邻接表做 BFS 或 DFS 是 O(V + E)：你访问每个顶点一次，走过每条边一次。同样的遍历在矩阵上是 O(V^2)，因为读取一个节点的邻居意味着扫描一整行 V 个格子，不管它们是不是边。在稀疏图上，这就是线性和平方的差别。底层容器的代价见[复杂度速查表](../complexity.md)。

## Python 实现

默认表示法是邻接表，`defaultdict(list)` 无需任何"这个键存在了吗"的样板代码就能建出一个：

```python
from collections import defaultdict, deque

def build_graph(edges, directed=False):
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        if not directed:
            graph[v].append(u)   # undirected: store both directions
    return graph

# weighted variant: store (neighbor, weight) tuples
def build_weighted(edges):
    graph = defaultdict(list)
    for u, v, w in edges:
        graph[u].append((v, w))
        graph[v].append((u, w))
    return graph
```

然后遍历它就是标准的 BFS：

```python
def bfs(graph, start):
    seen = {start}
    q = deque([start])
    order = []
    while q:
        node = q.popleft()
        order.append(node)
        for nxt in graph[node]:
            if nxt not in seen:
                seen.add(nxt)
                q.append(nxt)
    return order
```

**网格是一个隐式图。**你几乎从不为二维网格构建邻接表；边由四个（或八个）相邻格子隐含给出。`graph` 就是网格本身，而邻居函数就是边集：

```python
def neighbors(r, c, rows, cols):
    for dr, dc in ((1, 0), (-1, 0), (0, 1), (0, -1)):
        nr, nc = r + dr, c + dc
        if 0 <= nr < rows and 0 <= nc < cols:
            yield nr, nc
```

每一个"岛屿数量"、"腐烂的橘子"、"迷宫中的最短路径"问题都是一次图遍历，而你从不把图真正物化出来。认出网格是一个隐式图，就完成了一半的解答。

## 何时使用（以及何时不用）

在以下情况把问题建模为图：

- 实体之间有**任意的两两关系**：网络、依赖、地图、状态机。如果"A 与 B 相关"是核心，那它就是图。
- 你需要**可达性、连通性或路径**："我能从 X 走到 Y 吗"、"有多少个独立的组"、"最短路线"。
- 问题是一个**网格、一组课程先修关系，或单词变换**；这些都是伪装起来的图。

默认选择**邻接表**：它是 O(V + E) 空间，并给你廉价的邻居遍历，而这正是每次遍历所需要的。只有在图是**稠密的**（E 接近 V^2）、你需要频繁地做 O(1) 的"恰好这两个点之间有没有边"查询、或者 V 小到 O(V^2) 空间无关紧要且常数因子上的简洁性占优时，才选择**邻接矩阵**。

优劣一览：

| | 邻接表 | 邻接矩阵 |
|---|---|---|
| 空间 | O(V + E)，只存真实的边 | O(V^2)，始终如此，哪怕图很稀疏 |
| 边查询 `(u, v)` | O(degree(u))，扫描邻居列表 | O(1)，索引一个格子 |
| 遍历邻居 | O(degree(u))，只有真实邻居 | O(V)，扫描一整行 |
| 添加 / 删除边 | O(1) 添加；删除是 O(degree) | 两者都 O(1) |
| 最适合 | 稀疏图（大多数真实问题） | 稠密图、极小的 V、频繁的边测试 |
| 缓存 / 简洁性 | 更多指针追逐 | 扁平的二维数组，简单且缓存友好 |

## 权衡与陷阱

- **无向意味着存两个方向。**在列表里插入 `(u, v)` 和 `(v, u)`，或者在矩阵里同时设置 `m[u][v]` 和 `m[v][u]`。忘掉反向边是让你半张图变得不可达的经典 bug。
- **在入队时标记已访问，而不是出队时**，这是在 BFS 里。如果你等到弹出一个节点才标记它，同一个节点会被好几个邻居压入，你就会多次处理它，破坏 O(V + E) 的界限，有时还会破坏正确性。
- **`defaultdict` 在读取时创建键。**仅仅是对一个没有边的节点检查 `graph[node]`，就会插入一个空列表。通常无害，但它可能破坏后续的"这个节点在图里吗"测试。如果你绝不能改动它，就用 `.get(node, [])`。
- **矩阵的空间毫不留情。**一个有 100,000 个顶点的图需要一个 10^10 个格子的矩阵，这是不可能的，而它的邻接表也许只有几十万个条目。在大型稀疏图上，矩阵根本不是一个选项。
- **不连通的图需要一个外层循环。**从单个起点的一次 BFS 或 DFS 只能到达一个连通分量。要覆盖每个顶点（比如统计分量数），就遍历所有顶点，从每个未访问的顶点各发起一次遍历。

## 相关模式

- [图遍历](../patterns/16-graph-traversal.md)是在这个结构上的 BFS 和 DFS：连通性、连通分量、洪泛填充、网格问题。
