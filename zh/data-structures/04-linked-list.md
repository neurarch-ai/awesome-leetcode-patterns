# 04 - 链表

> 中文版。English: [04-linked-list](../../data-structures/04-linked-list.md)

链表把元素存储为一个个独立的节点，每个节点持有一个值和一个指向下一个节点的指针。没有连续的内存块，也没有索引：要到达第 k 个元素，你要从头节点走过 k 个指针。这单单一个事实，即没有随机访问，就是你所放弃的东西，而作为交换，你得到了在链表任意位置 O(1) 插入和删除的能力，前提是你已经持有那个节点。几乎每一道链表面试题实际上都是关于指针纪律：在你重接一个链接时，不要弄丢链表的其余部分。

![一个带快慢指针的单向链表](../../assets/linked-list.svg)

*一个单向链表。快慢指针用于找到中点或检测环。*

## 是什么

**单向链表** 是一条节点链。每个节点持有一个值和一个指向后继节点的 `next` 引用；最后一个节点指向 `None`。你保留一个指向第一个节点的 `head` 引用，那是你进入这个结构的唯一把手。要找任何东西，你都要从头节点顺着 `next` 指针走。

**双向链表** 给每个节点添加一个 `prev` 指针，这样你既能向前走也能向后走，并且你可以仅凭一个节点就删除它（你通过 `prev` 和 `next` 到达它两侧的邻居）。代价是每个节点要多存一个指针，且每次拼接时都要保持它一致。

其定义性属性是节点在内存中并非连续排布。没有任何算术能把索引变成地址，所以没有 O(1) 的 `a[i]`。你得到的是另一样东西：插入或删除一个节点不会移动任何其他元素，它只是重接几个指针。在数组中间插入是 O(n)，因为尾部必须滑动过去；链表的拼接是 O(1)，因为什么都不移动。

## 操作与复杂度

对于一个有 n 个节点的链表：

| 操作 | 复杂度 | 说明 |
|---|---|---|
| 访问 / 索引第 k 个节点 | O(n) | 从头节点走过 k 个 `next` 指针；无随机访问 |
| 查找一个值 | O(n) | 线性扫描，和无序数组一样 |
| 在头部插入（前插） | O(1) | 重接 `head`；什么都不移位 |
| 在尾部插入 | 有尾指针时 O(1)，否则 O(n) | 若必须走到尾部则是 O(n) |
| 给定节点做插入 / 删除 | O(1) | 全部意义所在：拼接两个指针 |
| 在位置 i 插入 / 删除 | O(n) | 被 O(n) 的走查主导以找到节点 |
| 仅给定节点删除（单向） | O(1) 技巧 | 把下一个节点的值拷进来再跳过它；对尾节点失效 |
| 仅给定节点删除（双向） | O(1) | `prev` 和 `next` 都在手 |
| 长度 | O(n) | 不存储；你要么数，要么另行追踪 |

主题是：变更本身是 O(1)，但 *定位* 节点是 O(n)。当一道题把节点直接交给你（或者你已经从上一步持有指向它的指针）时，你就获得 O(1) 的好处。当你必须先查找时，查找代价占主导，你又回到了 O(n)。关于它与 list 的比较，见 [复杂度速查表](../complexity.md)。

## Python 实现

Python 没有你在面试中使用的内置链表（`collections.deque` 底层是一个双向链表，但你从不触碰它的节点）。你要自己定义节点。规范的 `ListNode`：

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next
```

给定节点做插入和删除，这些是 O(1) 操作：

```python
def insert_after(node, val):
    # Splice a new node in after `node`. O(1), nothing else moves.
    node.next = ListNode(val, node.next)

def delete_after(node):
    # Remove node.next by skipping it. O(1).
    if node.next:
        node.next = node.next.next
```

**哑头哨兵**（dummy-head sentinel）是消除几乎每一种边界情况的技巧。与其对"万一我要删除头节点怎么办"或"万一链表是空的怎么办"做特殊处理，你在前面加一个一次性的节点，并始终在它后面一步操作。真正的答案是最后的 `dummy.next`：

```python
def remove_all(head, target):
    dummy = ListNode(0, head)   # sentinel sits before the real head
    prev = dummy
    while prev.next:
        if prev.next.val == target:
            prev.next = prev.next.next   # unlink; works even for the head
        else:
            prev = prev.next
    return dummy.next           # the head may have changed; this is safe
```

没有哨兵，你就需要为删除头节点单独设一个分支，因为它前面没有 `prev`。有了哨兵，前面就总是有一个。

原地反转单向链表是每场面试都期待的指针纪律练习，时间 O(n)、空间 O(1)：

```python
def reverse(head):
    prev = None
    curr = head
    while curr:
        nxt = curr.next     # save the rest before you overwrite the link
        curr.next = prev    # reverse this one pointer
        prev = curr         # advance both pointers
        curr = nxt
    return prev             # prev is the new head
```

`nxt = curr.next` 这一行是承重的：一旦你覆盖了 `curr.next`，你就弄丢了对链表其余部分的唯一引用，所以你必须先把它存起来。

## 何时用（何时不用）

**在以下情况用链表：**

- 你频繁在中间插入和删除，且你已经持有节点，从而获得 O(1) 拼接、无需移位。LRU 缓存是典型案例：双向链表提供 O(1) 的移到前端和逐出，哈希表提供 O(1) 的节点查找。
- 你在构建栈或队列，且想要保证两端 O(1)、无摊还重新分配。（在 Python 里你直接用 `deque`，它就是一个。）
- 题目字面上把节点交给你并要求你重接它们：反转、合并、检测环、重排。这些天生就是链表问题。

**在以下情况不要用链表：**

- 你需要随机访问或二分查找。索引是 O(n)，所以在链表上做二分查找每一步都是 O(n)，毫无意义。用数组。
- 你要大量迭代且在乎速度。节点散布在内存中，你得不到缓存局部性；尽管两者都是 O(n)，实践中数组遍历要快得多。
- 你只是需要一个可变长序列。Python 的 `list`（动态数组）更简单、更密集、追加是 O(1) 摊还。默认就选 list。

## 权衡与陷阱

**数组 vs 链表，诚实的对比。** 数组提供 O(1) 的索引访问和缓存友好的迭代，但在中间插入/删除是 O(n)，因为元素要移位。链表在给定节点时提供 O(1) 插入/删除，但访问是 O(n) 且局部性差。两者互不压倒：按你做得最多的操作来选。在面试中，只有当"给定节点 O(1) 拼接"这一属性是关键时链表才胜出（LRU 缓存、原地重排）；否则数组胜出。

**给定节点 O(1) 删除的技巧有个陷阱。** 在单向链表里，如果只给你一个要删的节点而没给它的前驱，你无法重接 `prev.next`。变通办法是把 *下一个* 节点的值拷进当前节点，然后删除那个下一个节点。它能做到 O(1)，但对尾节点会失效（没有下一个可拷贝），而且它是值拷贝，不是真正的解链。

**弄丢链表其余部分。** 头号 bug：在保存 `next` 指针所指之物之前就覆盖了它。在反转或重排时重新赋值前，永远先把 `curr.next` 存进一个临时变量。

**环。** 一个被破坏的或故意造成环的链表会让朴素遍历永远循环。用 Floyd 的快慢指针检测环（两个行走者，一个走两倍快；它们会在环内相遇），时间 O(n)、空间 O(1)。

**长度不是免费的。** 没有存储的计数。如果你需要长度，就走一遍链表（O(n)），或者在每次插入和删除时自己维护一个计数器。
