# 06 - 树与二叉搜索树

> 中文版。English: [06-tree-bst](../../data-structures/06-tree-bst.md)

树是一种层级结构：一个根节点带着若干子节点，每个子节点又是一棵子树的根，一路向下直到叶子。二叉树限制每个节点最多只有两个子节点。二叉搜索树（BST）在二叉树之上再加了一条排序规则，正是这一条规则把一堆无序的节点变成了一种支持 O(log n) 查找、插入和删除的结构，前提是它保持平衡。本文反复出现的主题是：每个 BST 操作的代价都是 O(height)，而高度究竟是 O(log n) 还是 O(n)，完全取决于平衡与否。

![展示排序不变量的二叉搜索树](../../assets/bst.svg)

*一棵 BST：左子树小于节点小于右子树，所以中序遍历的结果是有序的。*

## 它是什么

**树的术语**，因为面试中的词汇是精确的：

- **根（Root）**：唯一的顶端节点，没有父节点的那个。
- **叶子（Leaf）**：没有子节点的节点（一条分支的末端）。
- **边（Edge）**：父节点与子节点之间的连接。一棵有 n 个节点的树恰好有 n - 1 条边。
- **节点的深度（Depth）**：从根往下到该节点的边数。根的深度为 0。
- **节点的高度（Height）**：从该节点往下到某个叶子的最长路径上的边数。叶子的高度为 0。**树的高度**就是其根的高度。深度从顶往下数，高度从底往上数；它们在中间相遇，很容易混淆，所以要说清楚你指的是哪一个。
- **层（Level）**：所有深度相同的节点。第 0 层就是根。
- **平衡（Balanced）**：非正式地说，每个节点的左右子树高度相差不超过一个常数，因此高度保持在 O(log n)。**退化（degenerate）**的树则相反：每个节点都只有一个子节点，于是它实际上是一个高度为 n - 1 的链表。

**二叉树**允许每个节点最多有两个子节点（`left` 和 `right`），没有任何排序约束。它只是一个形状；你用 DFS 或 BFS 来遍历它。

**二叉搜索树**是二叉树加上 **BST 不变量**：对每个节点而言，其左子树中的每个值都小于该节点的值，其右子树中的每个值都大于它。（重复值通过固定约定处理，或者干脆不允许。）关键在于，这个不变量是递归的，作用于整棵子树，而不仅仅是直接子节点：左侧下两层的某个值仍必须小于其祖先，而不只是小于它的父节点。

## 操作与复杂度

设 h 为树的高度，n 为节点数。

| 操作 | 复杂度 | 说明 |
|---|---|---|
| 查找（BST） | O(h) | 比较后向左或向右下降；平衡时每一步丢弃一半 |
| 插入（BST） | O(h) | 查找空位，挂在那里 |
| 删除（BST） | O(h) | 找到它，然后剪接或与中序后继交换 |
| 求最小 / 最大（BST） | O(h) | 一路向左走（最小）或向右走（最大） |
| 前驱 / 后继（BST） | O(h) | 中序意义上的相邻节点 |
| 中序遍历 | O(n) | 每个节点访问一次；在 BST 上产生有序序列 |
| 任意完整遍历（DFS / BFS） | O(n) | 你必须触及每个节点 |
| 遍历所需空间 | O(h) | 递归栈（或显式栈 / BFS 队列） |

最重要的一句话：**BST 操作是 O(h)，而 h 完全取决于平衡。**

- **平衡**：h = O(log n)，因此查找、插入和删除都是 O(log n)。这就是你报出的那个数字，也是为什么平衡 BST 是"保持有序且快速更新"的首选结构。
- **退化**（把已排序的数据插入朴素 BST 恰好产生这种情况）：h = O(n)，于是每个操作退化到 O(n)，不比扫描链表更好。同样的代码，同样的不变量，但形状塌陷了。

### 为什么 BST 操作是 O(height)

查找从根开始，在每个节点上，BST 不变量让你能排除一整棵子树：如果你的目标小于当前节点，那它不可能在右子树的任何地方，所以你向左走，永远不看右边。每一步下降一层，因此步数受高度约束。当树平衡时，每一步还会把剩余候选减半（就像二分查找），得到 O(log n)。当树是一根退化的棍子时，"下降一层"只从考虑范围里移除一个节点，于是你又回到了 O(n)。

### 为什么 BST 的中序遍历是有序的

中序遍历是：先递归左子树，访问节点，再递归右子树。套用 BST 不变量：左子树里的一切都小于该节点，右子树里的一切都大于该节点。于是"所有更小的值，然后这个值，然后所有更大的值"，在每个节点上递归应用，就按非递减顺序输出这些值。这正是一整类问题背后的事实："BST 中第 k 小"、"验证 BST"（检查中序序列严格递增）以及"把 BST 转成有序链表"全都是中序遍历。

## Python 实现

标准的节点：

```python
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right
```

BST 查找，递归实现。每次调用凭不变量丢弃一侧：

```python
def search(root, target):
    if root is None or root.val == target:
        return root                 # None means not found
    if target < root.val:
        return search(root.left, target)   # target must be on the left
    return search(root.right, target)      # or the right
```

BST 插入，递归实现。像查找那样往下走；掉出去的地方就挂上：

```python
def insert(root, val):
    if root is None:
        return TreeNode(val)        # found the empty slot; create the node here
    if val < root.val:
        root.left = insert(root.left, val)
    elif val > root.val:
        root.right = insert(root.right, val)
    # val == root.val: duplicate; ignore (or handle per your convention)
    return root
```

中序遍历，它按有序顺序打印 BST：

```python
def inorder(root, out):
    if root is None:
        return
    inorder(root.left, out)         # all smaller values first
    out.append(root.val)            # then this node
    inorder(root.right, out)        # then all larger values
    return out
```

`search` 和 `insert` 都递归到至多为高度的深度，所以它们是 O(h) 时间和 O(h) 栈空间。`inorder` 每个节点访问一次，O(n) 时间，递归栈占 O(h) 空间。

## 何时使用（以及何时不用）

**在以下情况使用（平衡）BST：**

- 你需要数据保持有序，*并且*要快速插入和删除。数组保持有序时每次插入要 O(n)（搬移元素）；平衡 BST 用 O(log n) 就能做到。
- 你需要堆无法廉价回答的有序查询：前驱、后继、范围查询、"第 k 小"、下取整和上取整。在平衡 BST 上它们全是 O(log n) 或 O(h)。
- 你想要一个有序映射或有序集合。在面试中，当插入很少时，你常用一个普通有序列表加 `bisect` 来模拟它，因为 Python 并没有内置的平衡 BST。

**在以下情况不要使用 BST：**

- 你只是反复取单个最小或最大值。堆提供 O(1) 的查看以及更好的常数。
- 你需要按精确键做 O(1) 查找，且从不需要顺序。哈希表以 O(1) 平均复杂度胜过 O(log n)。
- 你无法保证平衡，而你的插入可能是已排序到来的。朴素 BST 会退化到 O(n)；要么用自平衡变体，要么换一种结构。

## 权衡与陷阱

**整个关键就在于平衡。**BST 所标榜的 O(log n) 其实是 O(h)，而 h 只有在树平衡时才是 O(log n)。给朴素 BST 喂入已排序的输入，它就会建成一条高度为 n 的直线，把每个操作都变成 O(n)。永远不要在没有"平衡"这个词的情况下给 BST 报 O(log n)。

**用范围来验证，而不是用父节点。**"这是不是一棵合法 BST"里的一个经典 bug 是：只检查每个节点的子节点相对于其直接父节点满足顺序。而不变量是作用于整棵子树的：向下传一个 `(low, high)` 范围并在每一步收紧它，或者检查中序遍历严格递增。

**删除是最琐碎的那个。**删除一个有两个子节点的节点，需要用它的中序后继（右子树中的最小值）或前驱来替换它，然后再删除那个后继/前驱。叶子和单子节点的删除很简单；两个子节点的情形才是 bug 藏身的地方。

**递归深度占 O(h) 空间。**树上的 DFS 不是 O(1) 空间：每个调用帧都压在栈上，所以你用 O(h) 空间，平衡时 O(log n)，退化时 O(n)。这正是[复杂度速查表](../complexity.md)里的递归空间警告；不要声称递归的树遍历是 O(1) 空间。

### 自平衡树（概览）

生产环境的有序结构不会相信输入会保持平衡；它们在每次插入和删除时重新平衡，以保证 h = O(log n)。你不会在面试里实现它们，但你应该能叫得出名字：

- **AVL 树**：严格平衡（每个节点的两棵子树高度相差至多 1），通过每次更新后的旋转来强制维持。查找最快；写入时旋转更多。
- **红黑树**：一种更宽松的平衡保证，通过节点"颜色"和旋转来强制维持。写入时旋转更少，树略高一些。这就是 Java 的 `TreeMap` 和 C++ 的 `std::map` 背后的东西。

共同的思路是：当一次插入或删除威胁到平衡时，常数次局部**旋转**会在 O(log n) 内恢复平衡，因此无论数据以什么顺序到来，高度、进而每个操作都保持对数级。Python 的标准库两者都不提供；实践中需要时你会转向 `sortedcontainers.SortedList`（一种不同的设计，但提供相同的 O(log n) 有序操作保证），或用 `bisect` 维护一个有序列表。
