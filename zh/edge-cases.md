# 边界情况清单

> 中文版。English: [edge-cases](../edge-cases.md)

因为没处理好边界情况而丢掉的 offer，比因为选错模式丢掉的还多。一个在正常路径上能跑、却在 `[]` 上崩溃的解法，会显得很粗心，而面试官恰恰会针对这些来试探。[解题框架](framework/solving-framework.md)把边界情况纳入了第 1 步（澄清）和第 6 步（测试）；本页则是一份汇总清单，供你在动手写代码前、以及在你说“做完了”之前，各跑一遍。

要点在于纪律：在澄清阶段就大声说出这些边界情况，把要紧的那些在函数开头处理掉，最后再把它们逐一走查一遍。早早说出这些还能体现资历，面试官会告诉你哪些在考察范围内。

## 覆盖大多数问题的五个

对每一道题都过一遍这份清单。这五个占了绝大多数陷阱。

1. **空输入。**`[]`、`""`、一棵空树、一个没有节点的图。你的循环是干净地跑零次，还是 `a[0]` 会抛异常？在写代码前就先定好空输入时的返回值：0、空列表、-1，还是抛一个错误。
2. **单个元素。**`[x]`、只有一个节点的链表或树。双指针和快慢指针的解法常常假设至少有两个元素；一个窗口或一个 `while left < right` 可能会跳过这唯一的元素。
3. **重复值。**重复的值会打破哈希集合逻辑里的唯一性假设，在[回溯](patterns/20-backtracking.md)和 [kSum](patterns/01-two-pointers.md) 里产生重复结果，还会改变“第 k 个不同的”这层含义。要问清输入能否包含重复，以及输出是否必须去重。
4. **负数（以及零）。**负数会打破[滑动窗口](patterns/02-sliding-window.md)在求和时背后的单调增长假设，使前缀和取余的技巧失效，并改变贪心的论证。零会破坏除法和“数组乘积”的逻辑。要明确问清取值范围。
5. **整数溢出。**一个超过 2^31 - 1 的和或积。在 Python 里无害（任意精度），但你仍然必须提到它，因为面试官在考察你是否知道它在 Java 或 C++ 里会咬人。要说“在定长语言里，我会在这里用 64 位类型或者取个模”。

## 按模式：那个专门咬你一口的边界情况

| 模式 | 要预先防住的边界情况 |
|---|---|
| [双指针](patterns/01-two-pointers.md) | 空或单个元素；两个指针交叉；重复值产生重复元组 |
| [滑动窗口](patterns/02-sliding-window.md) | 负数（单调收缩不再成立）；窗口比数组还大；不存在有效窗口 |
| [前缀和](patterns/03-prefix-sum.md) | 从下标 0 开始的子数组（需要一个值为 0 的前缀）；配合取余技巧时的负数 |
| [二分查找](patterns/07-binary-search.md) | 目标不存在；目标在第一个或最后一个下标；空数组；定长语言里 `(lo + hi)` 溢出 |
| [链表](patterns/10-linked-list.md) | 空链表；单个节点；有环；对头节点做操作（用一个哑节点） |
| [栈](patterns/11-stacks.md) | 从空栈里弹出；结尾处留下未匹配的括号 |
| [树](patterns/12-tree-dfs.md) | 根为空；单个节点；一棵倾斜（退化）的树使递归深度达到 O(n) |
| [图](patterns/16-graph-traversal.md) | 不连通的分量；自环；一个没有边的节点；没有 visited 集合导致重复访问 |
| [回溯](patterns/20-backtracking.md) | 输入中有重复需要排序并跳过；空子集；n = 0 |
| [DP](patterns/21-dp-linear-knapsack.md) | 基例（下标 0，容量 0）；无法到达的目标（干净地返回 -1 或无穷大） |
| [堆](patterns/24-heap.md) | k 大于输入规模；空堆；为了查看堆顶而弹出 |
| [数学](patterns/27-math.md) | 除以零；带取模的负操作数；把 0 和 1 作为特例 |

## 写代码前要问的关于输入形态的问题

把清单转成澄清性问题。在第 1 步就问这些，而不是等你已经造错了东西之后：

- 输入能是**空或 null** 吗？那时我返回什么？
- 会有**一个元素**吗？一个只有单节点的链表或树？
- 有**重复值**吗？输出必须唯一吗？
- 取值能是**负数、零或非常大**吗？确切的范围是多少？
- 输入是**有序的**、部分有序的，还是无序的？我能假设它有序吗？
- 对字符串：是 **ASCII 还是 Unicode**？区分大小写吗？能是只有空白字符吗？
- 对图：**连通**吗？**有环**吗？**有向**吗？会有自环或平行边吗？
- **保证存在一个有效答案**吗，还是说我必须发出“无解”的信号？

## 习惯

- **写代码前**，说出最可能击垮你思路的两三个边界情况，并先把边界检查写下来。
- **写完代码后**，把 `[]`、`[单个元素]`、一个全是重复值的输入和一个含负数的输入拿去走查代码。大多数 bug 会在这里冒出来，而不是在示例上。
- **即便在 Python 里也把溢出那句话说出来。**这是一个免费的信号，表明你会考虑语言边界，而它只花你一行。

边界情况不是你只对某些问题施加的一种模式。它是你对每一道题都要跑一遍的检查。那些从不被“万一数组为空呢”问倒的候选人，正是自己先把这个问题问出来的人。
