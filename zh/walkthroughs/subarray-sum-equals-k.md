# 精解：和为 K 的子数组 (LC 560)

> 中文版。English: [subarray-sum-equals-k](../../walkthroughs/subarray-sum-equals-k.md)

一个从头到尾在一道题上跑完六步框架的例题。目标是展示过程，而不只是答案。

## 题目

**LeetCode 560，中等。** 给定一个整数数组 `nums` 和一个整数 `k`，返回元素和恰好为 `k` 的连续子数组的总数。子数组由起始和结束下标定义，所以来自不同位置的 `[1, 2]` 和 `[2, 1]` 会分开计数。

例子：`nums = [1, 1, 1]`，`k = 2` 返回 `2`（下标 `0..1` 的子数组 `[1, 1]` 和下标 `1..2` 的那个）。

## 1. 厘清与复述

在动代码之前，我会大声问出这些问题：

- **输入类型。** `nums` 是一个整数列表。值可以是负数或零吗？这非常关键。如果全是正数，滑动窗口就能用，因为运行和是单调的。这道题允许负数和零，这就废掉了滑动窗口的想法，逼你换一种工具。除非另有说明，就假设允许负数。
- **我要返回什么？** 子数组的个数，不是子数组本身，也不是它们的下标。就是一个整数。
- **约束。** 典型上限是 `n` 到 `2 * 10^4`，值在 `[-1000, 1000]`，`k` 在完整的 int 范围内。解读 `n`：`2 * 10^4` 意味着 O(n^2) 解法大约是 `4 * 10^8` 次操作，这已经在过慢的边缘了。出题人是在暗示预期答案是 O(n)。
- **边界情况。** 空数组（答案 `0`）、单个元素等于 `k`（答案 `1`）、`k = 0` 且数组中有零（每个零都是它自己的一个有效子数组），以及负数让更长的前缀绕回到相同的和。

复述：数出和为 `k` 的连续切片的个数，数组可能包含负数，尽量做到 O(n)。

## 2. 手算一个例子

取 `nums = [3, 4, 7, 2, -3, 1, 4, 2]`，`k = 7`。

用眼睛看，和为 7 的子数组有：
- 下标 `0..1` 的 `[3, 4]`
- 下标 `2` 的 `[7]`
- 下标 `2..5` 的 `[7, 2, -3, 1]`
- 下标 `5..7` 的 `[1, 4, 2]`

一共四个子数组。注意第三个之所以存在，是因为那个 `-3` 把一段更长的序列又拉回到了 7。这恰恰是只支持正数的滑动窗口会漏掉的情形，也正是我需要一种能处理负数的技巧的原因。

## 3. 暴力解

显然的解法：固定每一个起始下标，向后延伸结束下标，维护一个运行和，每当它等于 `k` 就计数。

```python
def subarray_sum_brute(nums, k):
    count = 0
    n = len(nums)
    for start in range(n):
        running = 0
        for end in range(start, n):
            running += nums[end]      # sum of nums[start..end]
            if running == k:
                count += 1
    return count
```

两层嵌套循环，所以是 **O(n^2) 时间**和 **O(1) 空间**。正确，也是个不错的兜底，但在 `n = 2 * 10^4` 时有超时的风险。

## 4. 找到瓶颈并挑选模式

看看暴力解重复了什么。每换一个新的 `start`，它就从头重建运行和，重新走一遍之前为更早的起点已经加过的区域。被重复的工作是**移动窗口上的区间和**，而我没有利用的结构是：一个区间和可以写成两个前缀和之差。

把 `prefix[i]` 定义为 `nums[0..i-1]` 的和（所以 `prefix[0] = 0`）。子数组 `nums[j..i-1]` 的和就是 `prefix[i] - prefix[j]`。我想让这个差等于 `k`：

```
prefix[i] - prefix[j] == k   ->   prefix[j] == prefix[i] - k
```

所以在从左到右扫描并维护当前前缀和时，以位置 `i` 结尾、和为 `k` 的子数组个数，恰好等于此前值为 `prefix[i] - k` 的前缀和出现的次数。那是一次**重复查找**（「这个前缀值我见过多少次」），这直接指向一个**哈希表**，统计每个前缀和出现的频次。前缀和消掉了内层循环；哈希让查找变成 O(1)。

唯一的微妙之处：在扫描开始前用 `{0: 1}` 给哈希表播种，代表空前缀。正是它让从下标 0 开始的子数组能被计入（它的 `prefix[j]` 就是那个空前缀，值为 0）。

## 5. 写出代码

```python
from collections import defaultdict

def subarray_sum(nums, k):
    count = 0
    prefix = 0
    # seen[s] = how many prefixes so far have summed to s.
    # Seed with {0: 1} for the empty prefix, so subarrays that start at
    # index 0 are counted.
    seen = defaultdict(int)
    seen[0] = 1

    for x in nums:
        prefix += x                    # prefix sum of everything up to here
        # a subarray ending here sums to k when some earlier prefix == prefix - k
        count += seen[prefix - k]
        seen[prefix] += 1              # record this prefix for future ends

    return count
```

一趟扫描，一个字典。注意循环内的顺序：我在把当前前缀记录进去*之前*，就用哈希表给 `count` 加值。这可以防止在 `k = 0` 时把一个零长度的子数组算进去，因为在我查询时当前前缀还没进哈希表。

## 6. 测试、追踪与分析

追踪手算例子 `nums = [3, 4, 7, 2, -3, 1, 4, 2]`，`k = 7`。从 `seen = {0: 1}`、`prefix = 0`、`count = 0` 开始。

| x | prefix | prefix - k | seen[prefix - k] | count | 之后的 seen |
|---|--------|-----------|------------------|-------|------------|
| 3 | 3 | -4 | 0 | 0 | {0:1, 3:1} |
| 4 | 7 | 0 | 1 | 1 | {0:1, 3:1, 7:1} |
| 7 | 14 | 7 | 1 | 2 | {..., 14:1} |
| 2 | 16 | 9 | 0 | 2 | {..., 16:1} |
| -3 | 13 | 6 | 0 | 2 | {..., 13:1} |
| 1 | 14 | 7 | 1 | 3 | {..., 14:2} |
| 4 | 18 | 11 | 0 | 3 | {..., 18:1} |
| 2 | 20 | 13 | 1 | 4 | {..., 20:1} |

最终 `count = 4`，与我手算出的四个子数组吻合。`-3` 的情形被处理了：在 `x = 1` 时前缀又回到 14，而 14 之前见过一次，所以子数组 `[7, 2, -3, 1]` 被计入。

边界情况：
- **空数组**：循环从不执行，返回 `0`。正确。
- **单个元素等于 k**，`nums = [7], k = 7`：prefix 变成 7，`seen[0]` 是 1，count 是 1。正确。
- **零加上 k = 0**，`nums = [0, 0, 0], k = 0`：每一步都查询 `seen[prefix]`，而它已经保存了更早的零前缀的计数，于是累加得到 `1, 2, 3`，总共 6。这六个子数组正是三个零的所有连续切片。正确。

**复杂度：O(n) 时间**（一趟扫描，O(1) 的哈希表操作），以及在每个前缀都互不相同的最坏情况下哈希表所需的 **O(n) 空间**。这达到了约束所指向的目标，也胜过了 O(n^2) 的暴力解。

时间更充裕的话，我会补充：如果保证数组非负，滑动窗口就能把空间降到 O(1)；哈希表是支持负数所付出的代价。

## 面试官真正考察的是什么

你是否认得出「在可能有负数的情况下数出给定和的子数组个数」是一道前缀和加哈希的题，而不是滑动窗口。陷阱在于伸手去抓窗口，因为它是处理子数组问题的条件反射；而负数破坏了它的单调性。播种 `{0: 1}` 以及先查询后插入的顺序，是把内化了这个模式的候选人和只背下了它名字的候选人区分开来的两处细节。
