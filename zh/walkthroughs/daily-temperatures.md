# 精解：每日温度 (LC 739)

> 中文版。English: [daily-temperatures](../../walkthroughs/daily-temperatures.md)

一个从头到尾在一道题上跑完六步框架的例题。目标是展示过程，而不只是答案。

## 题目

**LeetCode 739，中等。** 给定一个每日温度的列表 `temperatures`，返回一个数组 `answer`，其中 `answer[i]` 是在第 `i` 天之后你需要等待多少天才能遇到更高的温度。如果之后没有更暖的一天，就填 `0`。

例子：`temperatures = [73, 74, 75, 71, 69, 72, 76, 73]` 返回 `[1, 1, 4, 2, 1, 1, 0, 0]`。

## 1. 厘清与复述

写代码前的问题：

- **输入类型。** 一个整数列表（温度，实际中通常在 30 到 100 之间，但我不会依赖这一点）。非空吗？就假设它可能只有一个元素。值可以重复，而「更暖」意味着严格更大，所以相等的温度不算。
- **我要返回什么？** 一个等长的列表，每一项是一个天数偏移量（不是下标，是距离），当之后没有更暖的一天时为 `0`。
- **约束。** 长度到 `10^5`。解读 `n`：`10^5` 让 O(n^2) 约 `10^10` 次操作，太慢。目标是 O(n)。
- **边界情况。** 只有一天（答案 `[0]`）、严格递减的温度（每个答案都是 `0`）、严格递增（除最后一个外每个答案都是 `1`）、全部相等（每个答案都是 `0`，因为相等不算更暖）。

复述：对每一天，用 O(n) 求出到下一个严格更暖的一天的距离。

## 2. 手算一个例子

`temperatures = [73, 74, 75, 71, 69, 72, 76, 73]`。

- 第 0 天 (73)：下一个更暖的是第 1 天 (74)。等 1。
- 第 1 天 (74)：下一个更暖的是第 2 天 (75)。等 1。
- 第 2 天 (75)：71、69、72 都更冷；第 6 天 (76) 更暖。等 4。
- 第 3 天 (71)：第 5 天 (72) 更暖。等 2。
- 第 4 天 (69)：第 5 天 (72) 更暖。等 1。
- 第 5 天 (72)：第 6 天 (76) 更暖。等 1。
- 第 6 天 (76)：之后没有更暖的。等 0。
- 第 7 天 (73)：最后一天，之后没有了。等 0。

结果 `[1, 1, 4, 2, 1, 1, 0, 0]`。注意第 2 天答案 4 是由它右边很远的一天解决的，而第 3、4 天都由同一个更暖的第 5 天解决。一天会一直「开着、等着」，直到某个更晚、更暖的一天把它关掉。

## 3. 暴力解

对每一天，向前扫描找到第一个更暖的一天。

```python
def daily_temperatures_brute(temperatures):
    n = len(temperatures)
    answer = [0] * n
    for i in range(n):
        for j in range(i + 1, n):
            if temperatures[j] > temperatures[i]:
                answer[i] = j - i
                break
    return answer
```

嵌套扫描，**O(n^2) 时间**，输出占 **O(n) 空间**。正确但在 `n = 10^5` 时太慢。

## 4. 找到瓶颈并挑选模式

暴力解一遍又一遍地重扫同样那些未来的日子。看看例子里的第 3、4 天：两者都向前扫，都落在第 5 天。那种向前重扫就是重复的工作。

结构性的观察：当我从左往右移动时，我会累积一些仍在**等待**更暖未来的日子。这些等待的日子，从最近往回读到最早，是按温度**递增**排列的。为什么？如果一个较早的等待日比一个较新的等待日更冷，那个较新（更暖）的日子早就该把它关掉了……不对：这个较新的日子无法关掉那个较早的，因为它并不比它更暖。所以这些等待的日子总是构成一个从最早到最新严格递减的温度序列，等价于从最新到最早递增。那就是一个**单调栈**。

机制：维护一个下标栈，栈里的下标其答案尚未确定，其温度沿栈往下递减。当新的一天到来时，它会解决栈顶所有比它更冷的等待日。每一个**被弹出的下标**都是一个「下一个更暖的一天」恰好是今天的日子，所以它的答案是 `today - popped`。然后把这新的一天压入栈，去等待它自己的更暖未来。每个下标压入一次、至多弹出一次，总共 O(n)。

## 5. 写出代码

```python
def daily_temperatures(temperatures):
    n = len(temperatures)
    answer = [0] * n
    stack = []                       # indices of days still waiting, temps decreasing

    for today in range(n):
        # today resolves every warmer-day-seeker colder than today's temp
        while stack and temperatures[today] > temperatures[stack[-1]]:
            prev = stack.pop()       # this earlier day's next-warmer is today
            answer[prev] = today - prev
        stack.append(today)          # today now waits for its own warmer day

    return answer                    # days left on the stack keep their default 0
```

不变式：任何时候栈中各下标处的温度都从栈底到栈顶严格递减。结束时仍留在栈上的任何下标都没找到更暖的一天，其 `answer` 保持初始化的 `0`，所以不需要对它们做任何特殊处理。

## 6. 测试、追踪与分析

追踪 `[73, 74, 75, 71, 69, 72, 76, 73]`。栈里存下标；我在括号里标出温度。

| today | temp | 弹出（已解决） | 设置的 answer | 之后的栈 |
|-------|------|-----------------|-----------|-------------|
| 0 | 73 | 无 | | [0] |
| 1 | 74 | 弹出 0 (73) | answer[0] = 1 | [1] |
| 2 | 75 | 弹出 1 (74) | answer[1] = 1 | [2] |
| 3 | 71 | 无 | | [2, 3] |
| 4 | 69 | 无 | | [2, 3, 4] |
| 5 | 72 | 弹出 4 (69)，弹出 3 (71) | answer[4] = 1, answer[3] = 2 | [2, 5] |
| 6 | 76 | 弹出 5 (72)，弹出 2 (75) | answer[5] = 1, answer[2] = 4 | [6] |
| 7 | 73 | 无 | | [6, 7] |

第 6、7 天留在栈上，answer 保持 `0`。最终 `[1, 1, 4, 2, 1, 1, 0, 0]`，与手算追踪吻合。注意第 5 天在一次到来中解决了两个等待日（先 4 后 3），而第 6 天解决了四个位置之前的第 2 天，正是暴力解靠重扫才付得起代价的那种远距离情形。

边界情况：
- **只有一天**，`[50]`：循环把下标 0 压栈，从不弹出，返回 `[0]`。正确。
- **严格递减**，`[80, 70, 60]`：从来没有更暖的，每天都被压栈、无一被弹出，返回 `[0, 0, 0]`。正确。
- **严格递增**，`[60, 70, 80]`：每天恰好弹出它前面那个，返回 `[1, 1, 0]`。正确。
- **全部相等**，`[70, 70, 70]`：`>` 是严格的，所以相等从不触发弹出，返回 `[0, 0, 0]`。正确。

**复杂度：O(n) 时间**（每个下标至多压入并弹出一次，所以内层 while 循环在整趟运行中总共做 O(n) 的工作，而不是每次迭代 O(n)），以及最坏情况下栈所占的 **O(n) 空间**（一个严格递减的输入会让每个下标都在等待）。这达到了目标，也胜过 O(n^2) 的暴力解。

时间更充裕的话，我会提一下同样的单调栈骨架，只要翻转比较和存储的内容，就能解决「下一个更大元素」、「下一个更小元素」和股票跨度问题。

## 面试官真正考察的是什么

你是否看出「下一个更暖的一天」是一道下一个更大元素的题，以及那个重复的向前扫描可以用一个存放未解决下标的栈来替换。要讲清楚的洞见是*一次弹出意味着什么*：当第 `today` 天到来时弹出下标 `prev`，正是你得知 `prev` 的答案的那一刻，因为 `today` 是第一个比它更暖的日子。能说出这个不变式、并论证摊还 O(n)（每个下标弹出一次）的候选人，是真正理解了这个模式，而不是套了个形状。
