# 解题框架

> 中文版。English: [solving-framework](../../framework/solving-framework.md)

编程面试奖励的是一个可重复的流程，而不是灵光一闪。强的候选人看起来很沉稳，是因为他们对每一道题都在跑同样的六个步骤，无论它简单还是难。内容在变；形态不变。把这些步骤大声地跑出来。面试官评的是你怎么想，而沉默会藏起你的思考。

这套流程对应一个 35 到 45 分钟的时段，还能留出富余。

## 1. 澄清并复述（2 到 3 分钟）

不要一上来就写代码。最快的失败方式，是自信地解错了题。先钉死这些：

- **输入和输出。**确切的类型和形态。数组是有序的吗？能是空的吗？取值是整数吗，能是负数或零吗？字符串是 ASCII 还是 Unicode？你到底要返回什么：值、下标、计数、布尔，还是所有解？
- **约束。**n 的规模是屋子里最重要的那个数字。它在你还没写下一行之前，就告诉了你目标复杂度。n 到 10 就是“暴力或回溯都行”。n 到几千允许 O(n^2)。n 到 10^5 或 10^6 意味着你需要 O(n log n) 或 O(n)。n 到 10^9 意味着你根本不会去遍历 n：要想到在答案上二分、数学，或位运算技巧。
- **边界情况。**空输入、一个元素、全是重复、全都相同、负数、溢出，以及“不存在有效答案”。现在就点出它们，这样它们才不会晚点来伏击你。

用你自己的话复述这道题，得到一个点头之后再设计。把假设写下来。

### 像编译器读一个类型那样去读 n

| 对 n 的约束 | 目标复杂度 | 它指向的模式 |
|---|---|---|
| n <= 12 左右 | O(2^n) 或 O(n!) | 回溯、位掩码 DP、暴力 |
| n <= 500 | O(n^3) | 区间 DP、Floyd-Warshall |
| n <= 5000 | O(n^2) | 成对 DP、两层嵌套循环 |
| n <= 10^5 | O(n log n) 或 O(n) | 排序、堆、二分查找、滑动窗口、双指针 |
| n <= 10^7 | O(n) 或 O(n log log n) | 单趟扫描、筛法、前缀和 |
| n 到 10^9+ | O(log n) 或 O(1) | 在答案上二分、数学、矩阵幂 |

这个约束是出题人递给你的一个提示。先读它。

## 2. 手动跑一个具体例子（2 分钟）

取一个小而不平凡的输入，在白板上手动解出来。这做三件事：它确认你理解了题目，它让边界情况浮出水面，而且它常常揭示出解法的结构。如果你连一个例子都手动解不出来，你就写不出通用情形。挑一个不退化的例子：不空、不已经排好序，且至少含一个有意思的情况。

## 3. 说出一个暴力解（2 分钟）

把那个显而易见、正确但慢的解法连同它的复杂度一起大声说出来。两个理由：它证明你理解了题目，而且它是你时间不够时的退路（一个能跑的暴力解胜过一个坏掉的聪明解）。“我可以检查每一对，那是 O(n^2)。让我看看能不能做得更好”正是那句该说的话。永远不要直接跳到最优解；你会丢掉流程分，还冒着在误解之上继续搭建的风险。

## 4. 找到瓶颈，然后伸手去够一个模式（5 分钟）

这是关键。看着暴力解，问：我在重复哪些工作，我没有利用哪些结构？

- **在一个移动的范围上重复工作**指向**滑动窗口**或**前缀和**：你在重新计算一个只在边缘发生变化的和或计数。
- **重复的查找**（“我见过这个值吗”）指向**哈希表**或**集合**，把查找变成 O(1)。
- **你在线性扫描的有序输入**指向**二分查找**或**双指针**。
- **递归里重复的子问题**指向**记忆化**，然后是**动态规划**。
- **“所有可能”或“每一种组合”**指向**回溯**。
- **“第 K 大”“top k”“合并 k 个”**指向**堆**。
- **一个单调的答案空间**（“可行的最小容量”）指向**在答案上二分**。

[速查表](../cheatsheet.md)是这套映射的完整版。本事在于把瓶颈的形态匹配到能消除它的模式。在写代码之前，说出你要伸手去够哪个模式，以及为什么。

## 5. 干净地写代码（10 到 15 分钟）

只有到现在你才动笔。读起来像资深的那些习惯：

- **给东西起有含义的名字**，而不是 `i2` 和 `tmp2`。用 `left`、`right`、`window_sum`、`seen`。
- **把你在第 1 步点出的边界情况**放在开头处理，而不是当作事后补丁。
- **一行只放一个想法。**面试官没法给一行他读不懂的代码评分。
- **边打字边说话。**把循环维持的不变量讲出来（“这次迭代之后，`window_sum` 就是 left 和 right 之间所有东西的和”）。

如果你卡住了，就回到暴力解并逐步优化它。一个部分正确、在改进中的解法是个强信号。

## 6. 测试、走查并分析（3 到 5 分钟）

代码写完不等于你做完了。当你已经说服面试官它是正确的，你才做完。

- **把你的手算例子**逐行走查一遍代码。这能抓到大多数 bug。
- **跑第 1 步的边界情况**：空、单个元素、全是重复、“无解”的情形。[边界情况清单](../edge-cases.md)是面试官会逐模式试探的完整列表。
- **说出最终复杂度**，时间和空间，并给出理由。如果它不是最优的，就明说，并说出最优解要花多少。
- **提一处你有更多时间会做的改进**（更好的空间、处理一个追问）。这表明你知道问题都埋在哪儿。

## 一段话版本

澄清输入、约束和边界情况；读 n 来猜目标复杂度。手动解一个例子。说出暴力解及其复杂度。找到重复的工作或未被利用的结构，并说出能消除它的模式。一边讲不变量一边干净地写代码。走查一个例子，跑边界情况，说出最终复杂度。然后停下来，喘口气。

## 人们输掉本可赢下的面试的常见方式

- **没澄清就写代码。**解错题是头号杀手。两分钟的提问能省下二十分钟的返工。
- **沉默。**如果你不出声，面试官就没东西可评，会往最坏处想。要出声地思考，哪怕卡住时也是。
- **追逐聪明解，结果什么都没落地。**一个能跑的 O(n^2) 胜过一个坏掉的 O(n)。先把暴力解落地，再改进。
- **忽视约束。**n 的规模是关于目标复杂度的一个免费提示。用它。
- **不测试。**把没走查过的代码交出去，还说“我觉得能行”，读起来很初级。走查它。
