# 04 - 哈希与频率计数
> 中文版。English: [04-hashing](../../patterns/04-hashing.md)

> **问题形态：** 「相加等于目标值的两个数，数组无序。」「这两个字符串是变位词吗？」「把变位词分组。」「第一个不重复的字符。」「这个数组含有重复元素吗？」凡是需要 O(1) 的成员判定、补数查找，或统计事物出现频率的问题。

哈希用空间换时间：一个哈希集合或哈希表把「x 在不在？」和「有多少个 x？」从 O(n) 的扫描变成 O(1) 的查找。它是无序数组问题背后的主力，也是当排序不被允许、或不值得那 O(n log n) 时你会伸手去拿的工具。

![对照已见集合做补数查找](../../assets/hashing.svg)

*一趟遍历，O(1) 查找：对每个 x，检查它的补数是否已在已见集合中。*

## 信号特征

看到以下情形时，考虑使用哈希集合或哈希表：

- **「我之前见过这个吗？」** 去重、对值做环检测、「含有重复」、「第一个重复元素」。集合以 O(1) 回答成员判定。
- **在无序数组上做补数或成对查找。** 无序 two-sum：对每个 `x`，问 `target - x` 是否已经见过。不用排序，O(n) 时间。
- **统计出现次数。** 变位词、「多数元素」、「第一个唯一字符」、「前 k 个高频」都归结为一张频率表。`collections.Counter` 是惯用写法。
- **按共享属性分组。** 变位词分组、按规范键分组。把每个项映射到一个键（排好序的字母、一个签名），再追加到以该签名为键的列表里。
- **任何「对每个元素扫描其余部分」的暴力解法**，其中那次内层扫描其实是一个成员判定或计数问题。那内层的 O(n) 会坍缩成 O(1)。

判断依据：数据没有可利用的顺序，问题问的是存在性、配对或频率，而不是区间或最优连续区间。

## 核心思想

哈希集合存储元素，通过把元素哈希到桶来以期望 O(1) 回答成员判定。哈希表存储键到值，并对按键查找做同样的事。你付出 O(n) 空间，换来平均情况 O(1) 的访问，正是这个折衷把平方级的成对扫描变成线性遍历。

补数技巧是最锋利的实例。在无序 two-sum 中你不需要同时拿到两个数：遍历数组，对每个 `x` 检查它的搭档 `target - x` 是否已在表中。若在，你就找到了这一对；若不在，记录 `x` 然后继续。每个元素被插入一次、被查询一次，因此整件事是 O(n) 时间、O(n) 空间，无需排序。

频率计数是另一半。一趟就建好从元素到计数的表，然后答案（最高频、唯一出现、匹配多重集）就是一次对该表的读取。变位词检查不过是「这两个字符串是否产生相等的计数表」，而分组则是「把所有产生相同键的东西收集到一起」。

**与双指针的对照。** [双指针](01-two-pointers.md) 靠利用顺序，以 O(n) 时间、O(1) 空间解决有序数组的成对问题。哈希在数组无序（或必须保留原始下标）时以 O(n) 时间但 O(n) 空间解决同一形态。这个选择是一个「空间对顺序」的折衷：如果输入已经有序，或者你无法花那额外空间，双指针胜出；如果它无序，而排序会破坏下标或代价太高，哈希胜出。

## 模板

**用已见集合做成员判定与去重：**

```python
# Time: O(n), Space: O(n)
def contains_duplicate(nums):
    seen = set()
    for x in nums:
        if x in seen:
            return True
        seen.add(x)
    return False
```

**补数表，无序 two-sum（返回下标）：**

```python
# Time: O(n), Space: O(n)
def two_sum(nums, target):
    pos = {}                          # value -> index
    for i, x in enumerate(nums):
        if target - x in pos:
            return [pos[target - x], i]
        pos[x] = i                    # record only after the check, avoids reusing i
    return [-1, -1]
```

**频率计数器，变位词检查：**

```python
# Time: O(n), Space: O(k) distinct characters (O(1) for a fixed alphabet)
def is_anagram(s, t):
    from collections import Counter
    return Counter(s) == Counter(t)   # equal multisets of characters
```

**按规范键分组，变位词分组：**

```python
# Time: O(n*k log k), Space: O(n*k)  (n words of length up to k)
def group_anagrams(words):
    from collections import defaultdict
    groups = defaultdict(list)
    for w in words:
        key = tuple(sorted(w))        # canonical signature shared by anagrams
        groups[key].append(w)
    return list(groups.values())
```

做分组时，诀窍在于选一个键，让它对恰好属于同一组的那些项完全相同：变位词用排好序的字母，若你想要 O(单词长度) 而非 O(单词长度 乘 log)，就用一个长度为 26 的计数元组。

## 变体

- **带计数的补数，而不只是存在性**（two-sum 中一个元素可能与相等的元素配对）。存储计数并检查补数的计数，处理 `x == target - x` 的自配对情形。
- **滑动窗口频率签名。** 「找出字符串中所有变位词」和「字符串的排列」是在一个运行计数表上、对照目标计数表的固定 [滑动窗口](02-sliding-window.md)。
- **最长连续序列。** 把所有元素放进集合，然后对每个在集合中没有前驱的数（一个序列的起点），向前行走计数。看起来像排序，实则 O(n)，因为每个元素恰被一个序列访问一次。
- **超越变位词的规范键分组。** 按位移模式给字符串分组、按斜率给点分组、按「排好序的数字」分组。模式是：设计一个范式，然后哈希它。
- **Counter 算术。** `Counter` 支持 `+`、`-`、`&`、`|` 和 `most_common(k)`，把「合并频率」、「多重集差」和「前 k 个高频」压缩成一行代码。
- **用定长字母表数组代替字典。** 当键是一个小的固定集合（小写字母）时，一个长度为 26 的列表就是更快的哈希表，常数因子是 O(1)，且没有哈希开销。

## 陷阱

- **在 two-sum 中先记录再检查。** 只在检验完补数之后才把 `x` 插入表中，否则单个元素会与自身配对，你会返回一个错误的 `[i, i]`。
- **可变或不可哈希的键。** 列表不可哈希；用作键之前转成 `tuple`。计数签名键必须是元组，不能是列表。
- **过期的零计数。** 在窗口中递减某个计数时，删除降到零的键，如果你基于 `len(map)` 或表相等来比较的话，否则一个零计数的键会让两个「相等」的多重集比较为不等。
- **假设 O(1) 是免费的。** 哈希花 O(n) 空间且有常数因子开销；对一个很小的固定字母表，数组更快，而如果空间是瓶颈，排序加 [双指针](01-two-pointers.md) 可能胜过它。
- **哈希顺序不是排序顺序。** 除了插入顺序，别依赖字典或集合的迭代顺序（Python 对字典保留插入顺序，对集合则不）。如果你需要有序输出，就显式排序。
- **自配对计数算错。** 在带计数的成对求和中，`x + x == target` 需要 `x` 至少出现两次，而不是一次。

## 延伸与相关模式

- 「数组是有序的，能省掉那额外空间吗？」会推向 [双指针](01-two-pointers.md)，它靠利用顺序以 O(1) 空间做成对查找。
- 「回答区间求和或子数组计数问题」会把这个与 [前缀和](03-prefix-sum.md) 结合：「前缀计数放进表里」的技巧就是把哈希应用到累积和上。
- 「随数据流入维护前 k 个」会推向 [堆](24-heap.md)；当你需要运行中的最优 k 个时，一张普通的频率表不够。
- 「不做完整排序找出 k 个最高频」会把频率表与 [top-k 与快速选择](09-top-k-quickselect.md) 配对。
- [滑动窗口](02-sliding-window.md) 里的窗口计数记账就是把这个模式应用到一个移动的区间上。
